Goal: Fix the complete coupon system on the pricing and checkout flow so that:
- Each plan has its own independent coupon input.
- A coupon applies only to the selected plan(s) it is configured for.
- The Stripe Checkout session is created with the correct discount.
- The "Invalid Coupon" error only appears when the coupon is truly invalid.

Very important:
- Do NOT touch or create the `shop_cart` table.
- Do NOT refactor unrelated modules or UI.
- Only work in coupon-related pricing UI, coupon validation logic, and Stripe checkout code.

------------------------------------------------
1) FRONTEND BEHAVIOR (Pricing page)
------------------------------------------------

Current bug:
- Typing coupon code (e.g., PREMIUM30) in one plan input automatically appears in all plan inputs.
- Free plan also shows coupon field although coupon may not be applicable.

Tasks:
1.1 Inspect the pricing page component (React) where all plan cards and the "Coupon code" inputs are defined.
     - Find how coupon state is stored (likely a single state shared across cards).
     - Refactor so that each plan card has its own local coupon state OR an indexed state per plan ID.

1.2 Make these changes:
     - When user types a coupon in the "Test Restricted Plan" box, it should NOT update other plans' inputs.
     - When the user clicks "Subscribe Now" on a plan, send only that plan's coupon code in the request payload 
       (e.g., { planId, quantity, couponCode }).

1.3 For the Free plan:
     - Either hide the coupon input for Free plan OR ignore the couponCode on the backend when plan is free.
     - UI should not mislead the user into thinking coupons apply to a free plan unless that is explicitly supported.

------------------------------------------------
2) BACKEND COUPON VALIDATION
------------------------------------------------

There is an "Invalid Coupon" error even when PREMIUM30 is active and configured in the dashboard.

Tasks:
2.1 Locate the backend route where coupons are validated and/or applied, for example:
     - /api/billing/validate-coupon
     - or in the same endpoint that creates the Stripe Checkout Session.

2.2 Understand the existing coupon model:
     - How coupons are stored (database table or config).
     - Fields: code, discountType, percentage/amount, startDate, expiryDate, usage limits, applicable plan IDs, status (Active).
     - This should correspond to the dashboard where coupon PREMIMUM30 has:
         - 30% discount,
         - active status,
         - start date 10/21/2025,
         - selected applicable plans (three checkboxes in the UI).

2.3 Implement correct validation logic:
     For a given { planId, couponCode }:
     - Check that coupon exists and status is Active.
     - Check today's date is within startDate and expiryDate (if expiry not null).
     - Check usage limits (total / per user) are not exceeded if those are enforced.
     - Check that the given planId is either:
          - included in coupon.applicablePlans, OR
          - if no plans selected, coupon is global.
     - If any of these fail, return a clear error, e.g.:
          { valid: false, message: 'Coupon not valid for this plan' }.

2.4 If the coupon is valid:
     - Return a structure with discount details, e.g.:
          { valid: true, discountType: 'percentage', percentage: 30, couponId: '<internal id or stripe coupon id>' }.

2.5 Make sure the current "Invalid Coupon" message on the frontend is driven by this backend logic, not a generic fallback.
     - If coupon is valid, frontend should not show the red "Invalid Coupon" banner.

------------------------------------------------
3) INTEGRATION WITH STRIPE CHECKOUT
------------------------------------------------

Tasks:
3.1 Find where Stripe Checkout Session is created (e.g., stripe.checkout.sessions.create).
     This should already be used when the user subscribes to any paid plan.

3.2 Modify the session creation logic so that:
     - If a valid coupon was provided and maps to a Stripe coupon or promotion code:
          - Attach it via `discounts: [{ coupon: 'stripe_coupon_id' }]`
            or `discounts: [{ promotion_code: 'stripe_promo_code_id' }]`
            depending on how coupons are stored.
     - If coupons are internal (only in our DB) and not Stripe coupons:
          - Either:
               a) Use a Stripe coupon that matches the same logic; OR
               b) Adjust our price/amount before sending to Stripe (if we already have that logic).
     - Keep existing behavior for when no coupon is provided.

3.3 Ensure that when coupon is not valid:
     - Checkout session is not created.
     - The API returns a 400 with a meaningful JSON error so the frontend shows "Invalid Coupon" correctly.
